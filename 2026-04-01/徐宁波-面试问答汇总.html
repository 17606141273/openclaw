二、专业技能 - 自动化测试框架

Q1: 详细讲讲你是如何搭建自动化测试框架的？

Web UI自动化（Selenium+Python+Pytest+YAML+POM）：

  • 1. POM模式：把页面元素和操作封装成Page类，比如LoginPage、HomePage，每个页面一个py文件；
  • 2. YAML管理测试数据：把环境配置、测试参数放到YAML文件，环境切换时只改配置不改代码；
  • 3. Pytest参数化：用@pytest.mark.parametrize实现一套用例多组数据，比如登录用例可以同时测正确账号、错误账号、空账号；
  • 4. conftest.py：共享fixture，比如登录初始化、driver初始化、截图fixture；
  • 5. 配合Jenkins：做持续集成，设置定时任务每天跑，失败自动发邮件。

Tesla接口自动化：

  • 1. 环境配置：配置测试环境、预发环境，区分测试数据和正式数据；
  • 2. 用例开发：按照业务模块分层编写用例，比如投放承接分为游戏、社交、电商三大模块；
  • 3. 物料库构建：把常用的请求参数、断言方法封装成可复用物料，减少重复开发；
  • 4. 持续集成：对接Jenkins，设置触发规则（代码提交/定时任务）；
  • 5. 结果分析：用Allure看报告，分析失败用例是环境问题还是代码问题。

我写了500+用例，覆盖核心回归场景，测试效率提升60%。

Q2: Pytest + YAML + POM 具体是怎么结合的？
  • 1. YAML：存放测试数据，比如登录测试的账号密码、接口测试的请求参数和预期结果；
  • 2. POM：页面元素定位和操作封装，比如一个LoginPage类，包含username输入框定位、password输入框定位、login按钮定位，以及输入账号、输入密码、点击登录三个方法；
  • 3. Pytest：测试用例层，调用POM的方法，用YAML的数据做参数化。

举例：登录测试用例，读取YAML中的3组账号数据（正确、错误、空），用@pytest.mark.parametrize驱动，分别调用LoginPage的方法执行，3组数据只用一套用例代码。

Q3: 自动化用例通过率95%是怎么做到的？

分层策略保证高通过率：

  • 1. 正向用例全覆盖：正常参数、正常流程的用例，100%覆盖，这类用例本身通过率就高；
  • 2. 边界异常重点覆盖：参数为空、类型错误、超长字符串等，提前在开发环境跑通，避免低级问题；
  • 3. 每日巡检：用Bits平台设置定时任务，每天跑一遍核心用例，发现问题及时修复；
  • 4. 失败用例快速响应：失败后立刻分析原因，是环境问题就找运维，是代码问题就提Bug给开发；
  • 5. 持续迭代：用例随着业务迭代不断优化，删除废弃用例，补充新场景。
Q4: 你觉得自动化测试的核心价值是什么？

核心价值是"提效"和"提质"：

  • 1. 提效：替代重复手动回归，比如每次发版前要跑100个用例，手动要2小时，自动化10分钟搞定；
  • 2. 提质：覆盖更多场景，手动测试受限于时间只能测核心路径，自动化可以24小时跑完整用例库；
  • 3. 快速反馈：开发提交代码后立刻跑用例，能在第一时间发现问题，降低修复成本；

但自动化不是万能的，适合：重复执行的用例、数据驱动型用例、边界条件多的用例。不适合：一次性探索性测试、UI经常变化的场景。

三、专业技能 - 接口测试

Q1: 接口测试如何保证覆盖率？

按四类场景设计用例，保证覆盖：

  • 1. 正向用例：正常参数、正常业务流程，验证功能正确性；
  • 2. 反向用例：参数缺失、参数类型错误、参数边界值（空字符串、特殊字符、超长）、必填参数省略；
  • 3. 异常用例：依赖服务不可用、网络超时、服务异常返回500，验证容错能力；
  • 4. 关联验证：上下游接口的数据传递是否正确，比如投放接口返回的广告ID，能否正确透传到落地页。

另外用Allure报告统计覆盖率，看看哪些模块用例少就补充。

Q2: 你常用哪些工具做接口测试？
  • 1. Tesla：字节内部平台，支持用例管理、参数配置、执行调度、报告生成；
  • 2. Python+Requests：用requests库发请求，Session管理会话，断言用assert或pytest；
  • 3. 辅助工具：Postman调接口、Fiddler抓包分析、Swagger看接口文档；
  • 4. 报告：Allure，支持用例分类、执行历史、失败截图、详细日志。
Q3: 如何处理接口依赖？
  • 1. 链路依赖：比如调用创建订单接口前需要先登录，我会用setup_class先调登录接口，保存token，后续接口复用；
  • 2. 数据依赖：用随机数或时间戳生成唯一标识，避免数据冲突；
  • 3. 业务流依赖：按业务顺序执行用例，后面的用例依赖前面用例的输出，用pytest的fixture传递；
  • 4. 如果依赖的接口不稳定：我会用mock服务模拟返回值，保证用例独立可运行。

Q2: Android和iOS兼容性测试有什么区别？

主要四大区别：

  • 1. 系统碎片化：Android多厂商ROM（MIUI、ColorOS、OriginOS），iOS统一，Android测试重点是多厂商适配；
  • 2. 权限管理：Android逐个授权+厂商自定义后台杀进程，iOS统一后台限制，测试重点是不同厂商权限策略；
  • 3. 推送通道：Android用厂商通道（华为/小米/vivo），iOS用APNs，测试重点是各通道送达率；
  • 4. 浏览器内核：Android用Chromium各版本，iOS用WebKit，Android更常出现CSS/JS兼容问题。

我的策略：Android覆盖华为/小米/OPPO/Vivo/荣耀各1-2台，iOS覆盖最新+老机型各1台。

Q3: 遇到过最有挑战的机型适配问题是什么？

印象最深的是华为和vivo的系统弹窗样式不一致问题：

  • 1. 问题：投放广告点击后，部分华为机型弹窗的确定按钮被系统状态栏遮挡，导致用户无法点击；
  • 2. 定位：用adb logcat抓日志，对比不同机型UI层级，发现是华为EMUI的弹窗布局和vivo OriginOS不同；
  • 3. 解决：协调开发针对不同厂商做UI适配，同时在自动化用例中增加厂商适配检查；
  • 4. 成果：这类问题拦截率80%+，减少上线后用户投诉。

五、专业技能 - 性能测试

Q1: 性能测试是怎么做的？

用JMeter工具，完整流程：

  • 1. 需求分析：明确测试目标，比如"秒杀接口支持1000并发，响应时间<2秒"；
  • 2. 场景设计：设置线程组（并发用户数）、Ramp-up时间（启动时间）、持续时间；
  • 3. 参数化：用CSV文件设置用户名密码等测试数据；
  • 4. 关联处理：提取sessionID、token等参数做关联；
  • 5. 断言：设置响应时间断言、响应内容断言；
  • 6. 执行与监控：运行测试，用Kepler看服务器CPU/内存/网络指标；
  • 7. 分析报告：看TPS、响应时间、错误率，分析瓶颈。
Q2: 性能瓶颈如何定位？

定位瓶颈的思路：

  • 1. 响应时间高：看是哪个环节耗时——网络传输？服务处理？数据库查询？
  • 2. 利用Kepler监控：看CPU、内存、磁盘IO、网络IO，找出资源瓶颈；
  • 3. 查日志：看慢查询日志、异常日志，定位具体接口；
  • 4. 逐层排查：从nginx→应用服务→数据库，逐层打点看哪个环节出问题；
  • 5. 之前项目：高峰期响应时间过长，通过监控发现是数据库查询慢，加索引后响应时间缩短50%。
Q3: 你认为性能测试最重要的是什么？

最重要的是"场景真实"和"指标明确"：

  • 1. 场景真实：模拟用户真实行为，不是凭空加压，比如秒杀接口要考虑秒杀开始前、进行中、结束后的不同流量；
  • 2. 指标明确：TPS、响应时间、并发数、错误率，这些指标要提前和业务方、开发确认合格标准；
  • 3. 瓶颈预判：提前识别可能的风险点，比如数据库、缓存、中间件的性能上限；
  • 4. 持续监控：性能问题不是测一次就完的，要建立基线，每次发版做回归对比。

六、专业技能 - 智能运维与容灾

Q1: 你是怎么做全链路监控的？

用字节Kepler智能运维平台：

  • 1. 接入监控：把业务涉及的服务（比如投放服务、落地页服务、广告服务）接入Kepler；
  • 2. 配置告警：设置阈值告警，比如接口响应时间>2秒、错误率>1%、服务宕机；
  • 3. 智能异常检测：开启智能异常检测，自动识别异常波动，不用人工设阈值；
  • 4. 链路追踪：看一个请求从端外到端内的完整调用链，哪个环节出问题一目了然；
  • 5. 多源数据关联：把日志、指标、追踪关联起来，快速定位根因。

大促场景特别有用，能在第一时间发现服务异常。

Q2: 混沌测试具体做什么？

用Chaos混沌平台模拟极端场景：

  • 1. 流量突增：模拟流量翻倍、瞬间峰值，看服务会不会挂；
  • 2. 依赖服务宕机：模拟广告服务、落地页服务不可用，验证调用方容错能力；
  • 3. 数据异常：模拟数据库连接失败、缓存失效，看系统表现；
  • 4. 网络延迟：模拟网络抖动、丢包，验证超时处理；

目的是验证系统容错和故障恢复能力，确保极端情况下用户体验不受影响。
