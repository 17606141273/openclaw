# 接口测试用例设计（含幂等性）

## 一、接口测试用例设计方法

### 1. 等价类划分
- 有效等价类：正常输入
- 无效等价类：异常输入（空值、边界值、错误类型）

### 2. 边界值分析
- 常见边界：0、空字符串、最大/最小值、超长输入

### 3. 场景分析法
- 正向场景：正常业务流程
- 逆向场景：异常流程（参数错误、顺序错误、数据不匹配）

---

## 二、接口测试用例核心检查点

| 检查点 | 示例 |
|--------|------|
| 参数必填 | 缺少必填参数时是否报错 |
| 参数类型 | 传错类型（如数字传字符串） |
| 参数边界 | 长度、范围边界 |
| 业务逻辑 | 状态流转、权限校验 |
| 异常处理 | 超时、网络错误 |
| 数据一致性 | 前后端数据对齐 |
| 安全性 | SQL注入、越权访问 |

## 三、接口幂等性测试

**幂等性**：同一接口多次执行，结果与单次执行一致

### 常见需要幂等的接口
| 类型 | 例子 | 风险 |
|------|------|------|
| 支付类 | 支付、扣款 | 重复扣款 |
| 订单类 | 创建订单 | 重复生成订单 |
| 积分类 | 签到、发放积分 | 重复发放 |
| 库存类 | 扣减库存 | 超卖 |

### 幂等性测试用例

#### 1. 参数一致性幂等
相同参数重复请求 → 返回相同结果
- 第一次：返回订单号 A
- 第二次：返回订单号 A（而非 B）

#### 2. 并发幂等
并发请求同一业务 → 结果一致
- 10个请求同时调用支付接口
- 结果：只扣款1次，10个请求都返回成功

#### 3. 异步回调幂等
第三方回调重复通知 → 只处理1次
- 第1次：更新订单状态为"已支付"
- 第2次：状态保持不变，不重复处理

#### 4. 重试幂等
接口超时重试 → 结果一致
- 超时后重试：返回相同退款结果

### 幂等性实现方案
| 方案 | 适用场景 | 原理 |
|------|----------|------|
| 唯一Token | 创建类接口 | 请求前获取唯一Token，提交时校验 |
| 唯一业务号 | 支付/退款 | 用订单号/流水号做幂等键 |
| 乐观锁 | 更新类接口 | 版本号/时间戳防重复 |
| 分布式锁 | 并发场景 | Redis锁限制并发 |
| 状态机 | 有状态流转 | 状态前置校验 |

### 幂等性测试检查点
- [ ] 相同参数重复调用 → 结果一致
- [ ] 并发相同请求 → 只生效1次
- [ ] 网络超时重试 → 不产生副作用
- [ ] 第三方回调重复 → 只处理1次
- [ ] 缓存+数据库一致性

## 四、登录接口用例示例（正向+异常）

### 正向用例
- 用户名+正确密码 → 登录成功

### 异常用例
- 用户名为空 → 提示"用户名不能为空"
- 密码错误 → 提示"密码错误"
- 用户名不存在 → 提示"用户不存在"
- 特殊字符注入 → 正常处理或拦截
